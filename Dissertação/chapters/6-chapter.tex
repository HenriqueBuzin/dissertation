\chapter{Testes}\label{cap:testes}
% ----------------------------------------------------------

Este capítulo descreve os procedimentos de teste adotados para avaliar a arquitetura proposta, abrangendo ambiente, cenários, métricas, coleta e análise dos dados.

% ----------------------------------------------------------
\section{Contextualização do Problema}
% ----------------------------------------------------------

Para avaliar a arquitetura proposta, foi adotado um cenário inspirado em aplicações de cidades inteligentes, com foco na gestão de recursos de água e energia. Nesse contexto, foram considerados medidores inteligentes que enviam dados de consumo para uma camada de névoa, onde ocorre o pré-processamento das informações antes de seu encaminhamento à nuvem.

Os dados utilizados para simulação foram obtidos a partir de conjuntos públicos, sendo um relacionado ao consumo residencial de energia elétrica \cite{uci_energy} e outro referente a medições de consumo de água \cite{greek_water}.

Antes da utilização nos testes, ambos os conjuntos de dados passaram por um processo de pré-processamento que incluiu a remoção de informações não essenciais e a agregação das medições em intervalos horários. Esse tratamento visou manter apenas os elementos relevantes para simular a operação contínua dos dispositivos, aproximando o comportamento dos dados ao fluxo esperado em um ambiente real de monitoramento urbano.

% ----------------------------------------------------------
\section{Ambiente de Testes}
% ----------------------------------------------------------

O ambiente de testes foi configurado para representar um cenário de cidade inteligente, no qual cada domínio de névoa corresponde a um bairro distinto. Cada névoa pode conter um nó primário, uma quantidade variável de nós de névoa, e um nó agregador.

A implantação e o gerenciamento desses componentes foram realizados por meio de uma \textit{interface web} desenvolvida para este trabalho, que interage diretamente com o Docker para criar e controlar as instâncias. Essa interface permite selecionar o bairro (névoa) no qual os componentes serão implantados, definir o tipo de contêiner e a quantidade de instâncias, além de oferecer controle individual ou em grupo para iniciar, pausar, parar e visualizar logs.

A Figura~\ref{fig:container_manager} apresenta a tela de gerenciamento de contêineres para o bairro Canasvieiras, localizado no município de Florianópolis, no estado de Santa Catarina. Nela, observa-se a criação de um nó de névoa e de um medidor de energia, ambos em execução. O painel agrupa as instâncias por tipo de componente, permitindo visualizar rapidamente seu estado e realizar ações sobre cada grupo ou instância individual.

\section{Tecnologias e Protocolos}
% ----------------------------------------------------------

Os experimentos consideram dispositivos medidores responsáveis pela geração de dados de consumo de água e energia. Esses dispositivos representam sensores de borda que enviam medições para a arquitetura proposta.

Para a geração dos dados utilizados pelos medidores, foram empregados conjuntos de dados públicos referentes ao consumo de energia elétrica e de água. O conjunto de dados utilizado para representar o consumo de energia elétrica foi obtido a partir de um dataset público disponibilizado pela UCI Machine Learning Repository\footnote{Disponível em: \url{https://archive.ics.uci.edu/dataset/235/individual+household+electric+power+consumption}.}. Já os dados utilizados para representar o consumo de água foram obtidos a partir de um dataset público disponibilizado pelo Hellenic Data Service\footnote{Disponível em: \url{https://data.hellenicdataservice.gr/dataset/78776f38-a58b-4a2a-a8f9-85b964fe5c95}.}. Ambos os conjuntos de dados passaram por um processo de pré-processamento antes de serem utilizados nos experimentos. Esse processo incluiu a padronização da frequência temporal dos registros para intervalos de uma hora, permitindo que os dados representassem um comportamento uniforme de envio de medições pelos dispositivos. Além disso, atributos e registros que não eram necessários para os experimentos foram removidos, mantendo apenas as informações relevantes para os testes realizados na arquitetura proposta.

Para avaliar a interoperabilidade entre os componentes da arquitetura, os
dispositivos medidores foram desenvolvidos para operar utilizando diferentes
protocolos de comunicação. Foram considerados os protocolos HTTP e CoAP,
permitindo observar o comportamento do sistema ao receber dados provenientes
de diferentes tecnologias de comunicação.

Para representar o funcionamento desses dispositivos no ambiente experimental,
foram desenvolvidos contêineres responsáveis por encapsular o comportamento dos
medidores. Foram implementados quatro contêineres distintos, correspondentes
às combinações entre tipo de medidor e protocolo de comunicação: medidor de
água via HTTP, medidor de água via CoAP, medidor de energia via HTTP e medidor
de energia via CoAP.

Os medidores baseados em \gls{HTTP} foram desenvolvidos em \textit{Python}, enquanto os medidores que utilizam o protocolo \gls{CoAP} foram implementados em \textit{Node.js}, devido ao melhor suporte e disponibilidade de bibliotecas específicas para esse protocolo no momento do desenvolvimento.

No ambiente de execução, o nó primário é responsável por receber dados utilizando os protocolos \gls{CoAP} e \gls{HTTP}. Independentemente do protocolo de origem, as mensagens são convertidas para \gls{HTTP} antes de serem encaminhadas aos nós de névoa, padronizando o tráfego interno do sistema.

Dentro de um nó de névoa, a comunicação entre a subcamada de protocolos e a subcamada de processamento ocorre por meio de \textit{WebSocket}, mantendo a conexão aberta e reduzindo o custo de abertura e fechamento contínuo de conexões. De acordo com Murley et al. \cite{murley2021}, conexões WebSocket permitem comunicação bidirecional persistente com menor overhead em comparação a abordagens baseadas em requisições HTTP repetidas, o que contribui para reduzir a sobrecarga e melhorar o tempo de resposta.

Após o processamento, os dados são estruturados em formato \gls{CSV}, a partir das mensagens recebidas em \gls{JSON}. A transmissão dos arquivos do nó de névoa para o nó agregador é realizada via \gls{SFTP}. O nó agregador consolida os arquivos provenientes de diferentes nós e os encaminha, também por \gls{SFTP}, à camada de nuvem para armazenamento na zona de entrada (\textit{landing zone}) do ambiente \gls{HPCC} \textit{Systems}.

\section{Distribuição entre Névoas e Balanceamento de Carga}
% ----------------------------------------------------------

Nos experimentos, os 70 dispositivos foram distribuídos entre dois domínios de névoa distintos, representando diferentes regiões no cenário de cidade inteligente.  
Cada tipo de dispositivo foi instanciado diversas vezes, totalizando 21 medidores de energia via \textit{HTTP}, 17 medidores de energia via \textit{CoAP}, 15 medidores de água via \textit{HTTP} e 17 medidores de água via \textit{CoAP}.  
Esses dispositivos transmitiram leituras reais de consumo em formato \gls{JSON}, com frequência horária, diretamente para a camada de névoa.

A distribuição inicial entre as névoas foi propositalmente desigual, a fim de avaliar o comportamento do sistema sob diferentes condições de carga:

\begin{itemize}
    \item \textbf{Névoa A}: 40 dispositivos  
    (12 medidores de energia via \textit{HTTP}, 10 via \textit{CoAP}, 9 medidores de água via \textit{HTTP} e 9 via \textit{CoAP}).
    \item \textbf{Névoa B}: 30 dispositivos  
    (9 medidores de energia via \textit{HTTP}, 7 via \textit{CoAP}, 6 medidores de água via \textit{HTTP} e 8 via \textit{CoAP}).
\end{itemize}

Quando uma das névoas atingia maior taxa de utilização, o mecanismo de balanceamento era acionado: o nó primário da névoa mais sobrecarregada redirecionava parte das requisições para a outra, realizando a conversão para \textit{HTTP} antes do envio, independentemente do protocolo de origem (\textit{HTTP} ou \textit{CoAP}).

Durante os testes, foram simuladas 48 \text{horas} de operação, resultando em um total de 3.360 leituras (70 dispositivos \text{ times } 48 envios cada), com medições horárias agregadas em janelas de consumo.  
No nó agregador, as medições de cada domínio de névoa foram consolidadas em um único arquivo por dia e transmitidas para o ambiente \gls{HPCC} \textit{Systems}, sem ocorrência de falhas durante a execução dos contêineres.

\section{Banco de Dados e Linguagem de Consulta de Dados}
% ----------------------------------------------------------

Nos experimentos realizados, foi utilizado um banco de dados local nos nós de
névoa para armazenamento temporário das informações recebidas dos dispositivos.
Esse armazenamento é utilizado durante o processamento local dos dados antes
de sua consolidação e encaminhamento para as camadas superiores da arquitetura.

Para essa finalidade, foi adotado o banco de dados \textit{MongoDB}. A escolha
desse banco se deu principalmente pela sua flexibilidade no armazenamento de
dados em formato \gls{JSON}, característica que facilita o tratamento das
mensagens provenientes dos dispositivos. Além disso, o \textit{MongoDB} permite
a inserção de registros heterogêneos sem a necessidade de um esquema rígido,
o que se mostrou adequado para lidar com diferentes tipos de medições e
estruturas de dados utilizadas nos testes.

A comunicação entre a subcamada de serviços e a subcamada de processamento foi
implementada utilizando a linguagem de consulta \textit{GraphQL}. Essa tecnologia
permite que as aplicações especifiquem exatamente quais campos e registros
devem ser retornados, reduzindo a transferência de dados desnecessários e
diminuindo a sobrecarga de processamento.

Outra característica relevante do \textit{GraphQL} é a possibilidade de realizar
consultas a múltiplas fontes de dados de forma unificada, agregando resultados
distintos em uma única resposta. Dessa forma, a subcamada de serviços consegue
solicitar apenas as informações necessárias ao processamento, recebendo uma
resposta já filtrada e adequada ao contexto da aplicação.

Cabe destacar que tanto o \textit{MongoDB} quanto o \textit{GraphQL} foram
utilizados como tecnologias de apoio na implementação dos testes. A arquitetura
proposta não depende especificamente dessas ferramentas, podendo utilizar
outras tecnologias equivalentes que atendam aos requisitos funcionais
definidos para o armazenamento e consulta de dados.

\section{HPCC Systems na Camada de Nuvem}
% ----------------------------------------------------------

Na fase de testes, os dados provenientes dos nós de névoa de cada domínio foram enviados para seus respectivos nós agregadores, cada um responsável de forma exclusiva pela consolidação das informações dentro do domínio ao qual pertence. Nesse processo, os arquivos no formato \texttt{.csv} gerados pelos nós de névoa foram unificados em um único arquivo consolidado. Quando necessário, o nó agregador realizou operações adicionais, como a soma de valores correspondentes.

Após a consolidação, cada nó agregador transmitiu o arquivo final para a camada de nuvem, implementada com o \gls{HPCC} \textit{Systems}. Os arquivos recebidos no formato \texttt{.csv}, que é um dos formatos suportados de forma nativa pela plataforma, foram armazenados e processados para análise posterior, representando o resultado da integração e do pré-processamento realizados nas camadas inferiores da arquitetura.

Esses arquivos foram disponibilizados na zona de entrada da plataforma (\textit{landing zone}), permanecendo prontos para processamento e consultas conforme as necessidades da aplicação.
