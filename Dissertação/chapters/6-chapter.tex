\chapter{Configuração e Execução dos Testes da Arquitetura IMSA}\label{cap:testes}
% ----------------------------------------------------------

Este capítulo descreve os procedimentos de teste adotados para avaliar a arquitetura \gls{IMSA}, abrangendo ambiente, cenários, métricas, coleta e análise dos dados.

% ----------------------------------------------------------
\section{Contextualização do Problema}
% ----------------------------------------------------------

Para avaliar a arquitetura \gls{IMSA}, foi adotado um cenário inspirado em aplicações de cidades inteligentes, com foco na gestão de recursos de água e energia. Nesse contexto, foram considerados medidores inteligentes que enviam dados de consumo para uma camada de névoa, onde ocorre o pré-processamento das informações antes de seu encaminhamento à nuvem.

Os dados utilizados para simulação foram obtidos a partir de conjuntos públicos, sendo um relacionado ao consumo residencial de energia elétrica \cite{uci_energy} e outro referente a medições de consumo de água \cite{greek_water}.

Antes da utilização nos testes, ambos os conjuntos de dados passaram por um processo de pré-processamento que incluiu a remoção de informações não essenciais e a agregação das medições em intervalos horários. Esse tratamento visou manter apenas os elementos relevantes para simular a operação contínua dos dispositivos, aproximando o comportamento dos dados ao fluxo esperado em um ambiente real de monitoramento urbano.

% ----------------------------------------------------------
\section{Metodologia de Avaliação}
% ----------------------------------------------------------

Após a implementação, foram realizados testes com foco em:
\begin{itemize}
    \item \textbf{Interoperabilidade}: avaliar a capacidade da \gls{IMSA} de lidar com diferentes protocolos de comunicação.
    \item \textbf{Balanceamento de carga}: avaliar o comportamento do nó primário na distribuição das requisições entre os nós de névoa, incluindo cenários de encaminhamento para outras névoas.
    \item \textbf{Agregação de dados}: analisar a consolidação realizada pelo nó agregador e seu impacto na redução de requisições à nuvem.
\end{itemize}

\section{Ambiente de Testes}
% ----------------------------------------------------------

O ambiente de testes foi configurado para representar um cenário de cidade inteligente, no qual cada domínio de névoa corresponde a um bairro distinto. Cada névoa pode conter um nó primário, uma quantidade variável de nós de névoa, e um nó agregador.

A implantação e o gerenciamento desses componentes foram realizados por meio de uma \textit{interface web} desenvolvida para este trabalho, que interage diretamente com o Docker para criar e controlar as instâncias. Essa interface permite selecionar o bairro (névoa) no qual os componentes serão implantados, definir o tipo de contêiner e a quantidade de instâncias, além de oferecer controle individual ou em grupo para iniciar, pausar, parar e visualizar logs.

Nela, observa-se a criação de um nó de névoa e de um medidor de energia, ambos em execução. O painel agrupa as instâncias por tipo de componente, permitindo visualizar rapidamente seu estado e realizar ações sobre cada grupo ou instância individual.

\section{Tecnologias e Protocolos}
% ----------------------------------------------------------

Os experimentos consideram dispositivos medidores responsáveis pela geração de dados de consumo de água e energia. Esses dispositivos representam sensores de borda que enviam medições para a arquitetura \gls{IMSA}.

Para a geração dos dados utilizados pelos medidores, foram empregados conjuntos de dados públicos referentes ao consumo de energia elétrica e de água. O conjunto de dados utilizado para representar o consumo de energia elétrica foi obtido a partir de um dataset público disponibilizado pela UCI Machine Learning Repository\footnote{Disponível em: \url{https://archive.ics.uci.edu/dataset/235/individual+household+electric+power+consumption}.}. Já os dados utilizados para representar o consumo de água foram obtidos a partir de um dataset público disponibilizado pelo Hellenic Data Service\footnote{Disponível em: \url{https://data.hellenicdataservice.gr/dataset/78776f38-a58b-4a2a-a8f9-85b964fe5c95}.}. Ambos os conjuntos de dados passaram por um processo de pré-processamento antes de serem utilizados nos experimentos. Esse processo incluiu a padronização da frequência temporal dos registros para intervalos de uma hora, permitindo que os dados representassem um comportamento uniforme de envio de medições pelos dispositivos. Além disso, atributos e registros que não eram necessários para os experimentos foram removidos, mantendo apenas as informações relevantes para os testes realizados na arquitetura \gls{IMSA}.

Para avaliar a interoperabilidade entre os componentes da arquitetura, os
dispositivos medidores foram desenvolvidos para operar utilizando diferentes
protocolos de comunicação. Foram considerados os protocolos HTTP e CoAP,
permitindo observar o comportamento do sistema ao receber dados provenientes
de diferentes tecnologias de comunicação.

Para representar o funcionamento desses dispositivos no ambiente experimental,
foram desenvolvidos contêineres responsáveis por encapsular o comportamento dos
medidores. Foram implementados quatro contêineres distintos, correspondentes
às combinações entre tipo de medidor e protocolo de comunicação: medidor de
água via HTTP, medidor de água via CoAP, medidor de energia via HTTP e medidor
de energia via CoAP.

Os medidores baseados em \gls{HTTP} foram desenvolvidos em \textit{Python}, enquanto os medidores que utilizam o protocolo \gls{CoAP} foram implementados em \textit{Node.js}, devido ao melhor suporte e disponibilidade de bibliotecas específicas para esse protocolo no momento do desenvolvimento.

No ambiente de execução, o nó primário é responsável por receber dados utilizando os protocolos \gls{CoAP} e \gls{HTTP}. Independentemente do protocolo de origem, as mensagens são convertidas para \gls{HTTP} antes de serem encaminhadas aos nós de névoa, padronizando o tráfego interno do sistema.

Dentro de um nó de névoa, a comunicação entre a subcamada de protocolos e a subcamada de processamento ocorre por meio de \textit{WebSocket}, mantendo a conexão aberta e reduzindo o custo de abertura e fechamento contínuo de conexões. De acordo com Murley et al. \cite{murley2021}, conexões WebSocket permitem comunicação bidirecional persistente com menor overhead em comparação a abordagens baseadas em requisições HTTP repetidas, o que contribui para reduzir a sobrecarga e melhorar o tempo de resposta.

Após o processamento, os dados são estruturados em formato \gls{CSV}, a partir das mensagens recebidas em \gls{JSON}. A transmissão dos arquivos do nó de névoa para o nó agregador é realizada via \gls{SFTP}. O nó agregador consolida os arquivos provenientes de diferentes nós e os encaminha, também por \gls{SFTP}, à camada de nuvem para armazenamento na zona de entrada (\textit{landing zone}) do ambiente \gls{HPCC} \textit{Systems}.

\section{Distribuição entre Névoas e Balanceamento de Carga}
% ----------------------------------------------------------

Nos experimentos, os 70 dispositivos foram distribuídos entre dois domínios de névoa distintos, representando diferentes regiões no cenário de cidade inteligente.  
Cada tipo de dispositivo foi instanciado diversas vezes, totalizando 21 medidores de energia via \textit{HTTP}, 17 medidores de energia via \textit{CoAP}, 15 medidores de água via \textit{HTTP} e 17 medidores de água via \textit{CoAP}.  
Esses dispositivos transmitiram leituras reais de consumo em formato \gls{JSON}, com frequência horária, diretamente para a camada de névoa.

A distribuição inicial entre as névoas foi propositalmente desigual, a fim de avaliar o comportamento do sistema sob diferentes condições de carga:

\begin{itemize}
    \item \textbf{Névoa A}: 40 dispositivos  
    (12 medidores de energia via \textit{HTTP}, 10 via \textit{CoAP}, 9 medidores de água via \textit{HTTP} e 9 via \textit{CoAP}).
    \item \textbf{Névoa B}: 30 dispositivos  
    (9 medidores de energia via \textit{HTTP}, 7 via \textit{CoAP}, 6 medidores de água via \textit{HTTP} e 8 via \textit{CoAP}).
\end{itemize}

Quando uma das névoas atingia maior taxa de utilização ou apresentava desequilíbrio na distribuição de especializações entre os nós disponíveis, o mecanismo de balanceamento era acionado. Nesse caso, o nó primário da névoa mais sobrecarregada redirecionava parte das requisições para a outra névoa, realizando a conversão para \textit{HTTP} antes do envio, independentemente do protocolo de origem (\textit{HTTP} ou \textit{CoAP}). Além do balanceamento entre diferentes domínios de névoa, o nó primário também é responsável por gerenciar a distribuição dos dispositivos entre os nós de névoa disponíveis dentro do mesmo domínio. Caso um nó apresente maior quantidade de dispositivos associados ou maior carga de processamento, o nó primário pode redistribuir parte desses dispositivos para outros nós da mesma névoa, buscando manter uma distribuição mais equilibrada dos recursos computacionais.

O sistema também monitora continuamente a disponibilidade dos nós de névoa. Caso um nó deixe de responder às verificações de conectividade, ele é automaticamente considerado indisponível e removido da lista de nós ativos. Nos experimentos realizados, o tempo máximo configurado para detecção de falha foi de 30 segundos. Após esse período sem resposta, o nó é removido do conjunto de nós disponíveis, evitando que novas requisições sejam direcionadas a ele.

Nos experimentos também foi considerado o cenário de falha de um nó primário. Os nós primários realizam verificações periódicas de disponibilidade a cada 30 segundos para confirmar que os demais domínios permanecem ativos. Caso um nó primário deixe de responder durante esse intervalo, ele é considerado indisponível, permitindo que o nó primário de outro domínio assuma temporariamente a coordenação. Nesse caso, são enviadas mensagens de controle aos dispositivos conectados, instruindo-os a redirecionar suas transmissões para o novo nó primário disponível. Esse mecanismo permite manter o fluxo de dados mesmo na ausência do nó primário originalmente responsável pelo domínio.

Durante os testes, foram simuladas 48 horas de operação, resultando em um total de 3.360 leituras (70 dispositivos $\times$ 48 envios cada), com medições horárias agregadas em janelas de consumo. No nó agregador, as medições de cada domínio de névoa foram consolidadas em um único arquivo por dia e transmitidas para o ambiente \gls{HPCC} \textit{Systems}, sem ocorrência de falhas durante a execução dos contêineres.

\section{Tecnologias de Armazenamento e Consulta de Dados na Arquitetura}
% ----------------------------------------------------------

Nos experimentos realizados, foi utilizado um banco de dados local nos nós de névoa para armazenamento temporário das informações recebidas dos dispositivos. Esse armazenamento é empregado durante o processamento local dos dados, antes de sua consolidação e encaminhamento para as camadas superiores da arquitetura.

Para essa finalidade, foi adotado o banco de dados MongoDB. A escolha dessa tecnologia está relacionada à sua flexibilidade na manipulação de dados em formato \gls{JSON}, o que facilita o tratamento das mensagens provenientes dos dispositivos. Além disso, a ausência de um esquema rígido permite a inserção de registros heterogêneos, característica importante para lidar com diferentes tipos de medições e estruturas de dados utilizadas nos experimentos.

A comunicação entre a subcamada de serviços e a subcamada de processamento foi implementada utilizando \gls{GRAPHQL}. Nesse contexto, sua utilização possibilita a recuperação eficiente apenas dos dados necessários para cada operação, contribuindo para a redução do volume de informações trafegadas e da sobrecarga de processamento.

Além disso, o uso de \gls{GRAPHQL} permite a integração de diferentes fontes de dados de forma unificada, possibilitando que a subcamada de serviços realize consultas mais específicas e receba respostas já estruturadas conforme as necessidades da aplicação.

Cabe destacar que tanto o MongoDB quanto o \gls{GRAPHQL} foram utilizados como tecnologias de apoio na implementação dos experimentos. A arquitetura \gls{IMSA} não depende especificamente dessas ferramentas, podendo ser adaptada para a utilização de outras tecnologias equivalentes, conforme os requisitos específicos de cada aplicação.

\section{HPCC Systems na Arquitetura}
% ----------------------------------------------------------

O \textit{High Performance Computing Cluster} (\gls{HPCC}) \textit{Systems} é uma plataforma de processamento massivamente paralelo, de código aberto, desenvolvida pela divisão \gls{HPCC} Systems da LexisNexis Risk Solutions. Seu objetivo é lidar com grandes volumes de dados, desde a ingestão até a entrega de produtos de dados, com tempos de resposta reduzidos. Criado antes da popularização do Hadoop, não se baseia no paradigma \textit{map-reduce}, possuindo uma arquitetura própria \cite{taylor2022}.

A plataforma é composta por um conjunto de servidores responsáveis por funções como compilação de código, gerenciamento distribuído de dados e execução de trabalhos (\textit{workunits}). O processamento é realizado por dois tipos de clusters: Thor e \gls{ROXIE}.

O Thor é voltado para operações intensivas em grandes conjuntos de dados, realizando tarefas como transformação, limpeza, integração e análise em larga escala. Já o \gls{ROXIE} é destinado a consultas rápidas e atendimento a solicitações de usuários finais.

Ambos os clusters utilizam a linguagem \gls{ECL}, de natureza declarativa, permitindo que o sistema determine a forma mais eficiente de execução das operações.

No contexto da \gls{IMSA}, o \gls{HPCC} \textit{Systems} é utilizado na camada de nuvem como infraestrutura para armazenamento e processamento de dados em larga escala. Durante os experimentos, os dados provenientes dos nós de névoa foram consolidados pelos nós agregadores e transmitidos para o ambiente \gls{HPCC} \textit{Systems}.

Os arquivos gerados no formato \texttt{.csv} foram armazenados na \textit{landing zone} da plataforma, permanecendo disponíveis para processamento e análise conforme as necessidades da aplicação.
